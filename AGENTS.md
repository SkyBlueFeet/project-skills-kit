# Repository Guidelines

## 项目定位

本仓库是项目规则与协作文档的模板源仓库，用于协助规范制定、模板维护与 CLI 开发。

- 规范模板的权威内容位于 `templates/base/`
- 根目录文件只描述本仓库自身的维护方式
- 目标项目需要的 `developers/**` 文档由 CLI 从 `templates/base/` 复制生成

---

## 维护入口

- 仓库说明：`README.md`
- 接入说明：`ONBOARDING.md`
- 模板入口：`templates/base/AGENTS.md`
- 会话留痕：`developers/SESSIONS/`

---

## 规范目标

维护本仓库中的模板与 CLI 时，默认以以下目标作为最终判断标准：

1. 让开发者能够快速看懂模块、业务逻辑与代码实现，并定位相关文件。
2. 让 Agent 能快速理解项目背景、技术栈、文档结构、规范要求与代码位置，并在改动后完成必要留痕。
3. 为 TypeScript、JavaScript、Java、Python、HTML、CSS 等语言提供可落地的代码规范与协作约束。
4. 在满足上述目标的前提下，坚持按需加载与最小读取集，控制上下文成本。

如无更具体约束，涉及模板、规则、索引、技能、CLI 的改动，应优先判断它是否更接近上述四个目标。

---

## 维护原则

- 修改规范文档时，优先改 `templates/base/**`，不要在根目录保留第二份同构副本。
- 根目录 `AGENTS.md`、`README.md`、`CLAUDE.md` 只保留模板制定、CLI 开发与仓库维护所需的最小说明。
- 未明确要求时，不主动改动 `docs/` 下的用户文档。
- 每次任务结束后，追加 `developers/SESSIONS/NOTE_YY_MM_DD.md` 留痕。
- 涉及计划状态变化时，只更新本文件中的计划索引。
- 对外接口或模板结构变更时，优先保证 CLI 与 `ONBOARDING.md` 一致。

## 本仓库 JavaScript 开发约束

> 本节**仅约束当前仓库自身的 JavaScript 开发**，主要适用于 `src/**/*.js`。

- 模块职责保持单一：`src/cli.js` 只做启动，命令分发放在 `src/cli/main.js`，具体逻辑放在 `src/cli/commands/*.js`，共享能力放在 `src/cli/*.js` 或 `src/cli/utils/*.js`。
- 新增逻辑优先继续按现有模块边界拆分，避免把参数解析、文件 I/O、迁移逻辑、诊断逻辑重新堆回单文件。
- 统一使用 ES Modules；路径处理必须使用 Node.js `path` API，不写死 Windows 或 Unix 路径分隔符。
- 需要持久化或对外输出的相对路径，统一归一化为 `/` 风格，避免跨平台差异。
- 导出的函数、共享常量、命令分发表、配置映射、持久化数据结构，如存在边界条件、隐含约束或复用价值，必须补 JSDoc。
- 复杂对象结构优先用 JSDoc `@typedef` 描述，避免长期保留难以理解的 `Record<string, any>` 风格约定。
- 注释默认解释“为什么这样做”或“这里的约束是什么”，不复述代码表面行为；局部临时变量通常不写注释。
- 涉及兼容处理、迁移逻辑、路径归一化、锁文件结构、失败分支时，应补简洁说明性注释，帮助后续维护者快速判断改动边界。
- 新增或修改 CLI 行为后，至少执行一次相关命令冒烟或语法校验；如无更合适命令，最低基线为 `node --check` 与 `node ./src/cli.js --help`。
