# 代码规范（工程执行版）

> 目标：提高可维护性、可测试性与协作效率。适用于 `src/` 与后续新增模块。

## 1. 模块化与单一职责

- 一个文件只负责一个明确能力域，不混合 CLI、配置、业务编排、适配器实现。
- 一个类/函数只做一件事；若函数出现“解析 + 校验 + 执行 + 输出拼装”四类动作，必须拆分。
- 新能力优先新增模块，不在现有大文件中堆叠分支。
- 目录分层必须稳定：`core`（抽象与协议）、`runtime`（编排）、`providers`（适配器）、`config`（配置加载）、`tools`（工具实现）。

## 2. 面向对象设计要求

- 对外能力必须经接口抽象暴露，禁止业务代码直接依赖具体 provider 类。
- 优先使用“组合”而非继承；继承仅用于稳定且可复用的模板流程。
- 类职责必须可描述为一句话；无法一句话描述时，需要拆分类。
- 对象创建逻辑与业务逻辑分离：工厂只负责创建，runner 只负责编排。

## 3. 单文件行数约束

- 强约束：单文件不超过 250 行（测试文件不超过 300 行）。
- 预警线：超过 180 行必须评估拆分方案，并在 PR 描述写明是否拆分及原因。
- 新增代码不得把已超限文件继续放大；应优先先拆再扩展。

## 4. 注释规范（中文）

- 注释语言统一为中文，说明“为什么这样做”，不是复述代码。
- 必须写注释的位置：
  - 边界条件与异常分支
  - 协议映射与兼容逻辑
  - 有副作用或性能权衡的代码
- 导出的函数、类型、接口、共享常量、配置映射，若存在复用边界或隐含约束，必须补 JSDoc 或等价文档注释。
- 文档注释至少说明：用途、关键参数/返回值、字段含义、失败行为；对象结构优先用 `type/interface` 或 `@typedef` 明确表达。
- 禁止无信息注释：如“给变量赋值”“调用函数”。
- 注释率要求：核心源码（`src/**/*.ts`，排除 `*.test.ts`）目标不低于 8%。

## 5. 健壮性与可测试性

- 所有外部输入必须显式校验：CLI 参数、env、配置文件、模型/tool 返回值。
- 错误信息必须可定位（包含字段名/模型 id/文件路径）。
- 每个新增能力至少包含 1 条成功路径 + 1 条失败路径测试。
- 对关键编排流程定义失败兜底策略，不允许静默吞错。

## 6. 命名与可读性

- 文件名使用 `kebab-case`；导出优先具名导出。
- 函数名使用动词短语，类名使用名词短语，布尔值前缀统一 `is/has/can/should`。
- 单个函数建议不超过 40 行；超过 60 行必须拆分。
- 条件分支超过 3 层时，应引入策略对象或早返回重构。

## 7. 执行机制（必须）

- CR 检查项必须包含：职责边界、文件行数、中文注释、失败路径测试。
- 若违反强约束（行数上限、无失败路径测试、无输入校验），默认不合并。
- 技术债允许临时豁免，但必须在同次变更中记录整改计划与截止时间。
- 功能完成基线：`npm run lint` 与 `npm run test:all` 必须同时通过。
- 推荐统一门禁命令：`npm run quality:gate`（串行执行 lint + tests）。
- 代码质量巡检命令：`npm run quality:check`（生成 `developers/REPORTS/CODE_QUALITY_REPORT_*.md` 报告）。
