## 一、功能目标核对

对照计划文档，逐条确认以下问题：

- [ ] 计划中声明的所有 API 端点均已实现，路由、方法、路径与设计一致
- [ ] 各端点的请求参数校验（必填、类型、边界）已实现
- [ ] 各端点的响应结构与约定格式（字段、状态码）一致
- [ ] 数据库操作（增删改查）覆盖计划要求的业务场景
- [ ] 业务逻辑的核心分支（正常路径 + 异常路径）均已实现
- [ ] 对外依赖（第三方 API、消息队列、缓存）的集成符合计划要求

## 二、代码质量检查

```bash
# 格式化与静态检查
npm run lint          # ESLint / Biome / 项目配置
npm run format:check  # Prettier 检查（如有）

# 单元测试与集成测试
npm run test          # Jest / Vitest / Mocha

# 构建验证（如为 TypeScript 项目）
npm run build
```

**判定要求：**
- lint 无 error（warning 可记录但不阻断）
- 与计划目标直接相关的测试用例全部通过
- TypeScript 项目构建无类型错误

## 三、非功能检查

### 3.1 接口健壮性

- [ ] 所有入参均有校验，非法输入返回明确错误（4xx），不暴露内部信息
- [ ] 必要的幂等性设计已落地（如 POST 创建类接口）
- [ ] 分页、排序、过滤参数有默认值与上限约束

### 3.2 错误处理与日志

- [ ] 全局错误处理中间件已配置，异常不会裸出堆栈到客户端
- [ ] 关键操作（创建、删除、外部调用）有结构化日志记录（包含 traceId / requestId）
- [ ] 预期内的业务错误（如资源不存在）与非预期错误（500）的日志级别有区分

### 3.3 安全基线

- [ ] SQL / NoSQL 查询使用参数化/ORM，无拼接风险
- [ ] 敏感配置（密码、密钥）读自环境变量，未硬编码
- [ ] 鉴权中间件覆盖了需要保护的路由（如有鉴权需求）
- [ ] 依赖项无已知高危漏洞（`npm audit --audit-level=high`）

### 3.4 性能基线

- [ ] 数据库查询对高频访问字段有索引（结合计划中的查询场景）
- [ ] 无明显的 N+1 查询问题
- [ ] 长耗时操作（大文件、批量处理）使用异步/流式处理，不阻塞事件循环

### 3.5 可运维性

- [ ] 健康检查接口已实现（`/health` 或等价）
- [ ] 优雅退出（SIGTERM）已处理，连接可正常关闭
- [ ] 环境变量有文档（`.env.example` 存在且与实际一致）
