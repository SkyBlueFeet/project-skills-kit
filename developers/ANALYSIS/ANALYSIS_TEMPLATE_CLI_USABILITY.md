## 1. 背景与范围

本分析仅评估两部分内容：

- `templates/base/**`：目标项目实际接入后会使用的规范模板。
- `src/cli/**`：负责初始化、补装、同步、迁移、诊断这些模板的 CLI 实现。

评估角度只看“实际使用是否顺手、是否一致、是否能长期执行”，不展开根目录说明文件本身的问题。

## 3. 总体结论

当前模板体系已经具备比较清晰的结构：入口、技能、规范、留痕、计划、验收之间的分层关系是成立的；CLI 也已经覆盖了初始化、补装、迁移、同步、诊断这些关键动作。

但从实际使用角度看，当前最大问题不是“模板写得不全”，而是“CLI 还没有把模板中的模式差异和治理强度完整落地”。结果就是：

1. 模板里声明了轻重分层，但 CLI 实际更接近“默认全量安装”。
2. `skills.lock.json` 已经存在，但还没有真正成为同步和诊断的单一配置真源。
3. 少数高成本规则（尤其代码索引）仍停留在文档层，自动化支撑不足。

这会影响两个关键体验：

1. 新项目第一次接入时感觉偏重。
2. 接入后随着项目演进，模板、锁文件、CLI 行为容易逐渐失配。

---

## 4. 主要问题

### 问题 1：文档治理开关不是一个真正可落地的模式

**优先级**：高

**现象**：

- `init` 在 `includeDocsGovernance=false` 时会删除 `developers/SKILLS/`、`developers/SESSIONS/` 和 `MODULE-BUSINESS-FILE-MAP.md`。
- 但模板 `AGENTS.md` 仍要求任何任务先读 `developers/SKILLS/SKILL_ROUTER.md`，并要求完成后写 `SESSIONS`。
- `developers/INDEX.md` 仍把 `SKILLS`、`SESSIONS`、`MODULE-BUSINESS-FILE-MAP` 当作常规入口。
- `doctor` 还把 `developers/SKILLS/SKILL_ROUTER.md` 视为必需文件。

**影响**：

- 关闭文档治理后，生成的项目会立即出现“模板入口要求存在，但 CLI 又删除了它”的自相矛盾状态。
- 团队无法把它当成真正的轻量模式使用。

**建议**：

1. 明确产品化成两种或三种模式，例如 `lite / standard / full`。
2. 不同模式使用不同模板入口内容，而不是先全量复制再删目录。
3. `doctor` 和 `sync` 必须基于模式判断哪些文件是必需项。

### 问题 2：`init` 的默认策略仍偏“全量下发”，与按需加载理念不完全一致

**优先级**：高

**现象**：

- `init` 先把 `templates/base` 整体复制到目标目录。
- 后续只做有限裁剪：是否保留 `CLAUDE.md`、是否保留文档治理目录、是否保留所选语言规范。
- `skills.lock.json` 里会登记模板技能目录中的大量技能文件。

**影响**：

- 目标项目第一次接入就拿到一整套偏重的治理结构。
- 模板自己强调的“最小读取集”“按需加载”更多体现在阅读原则，而不是安装策略。
- 团队更容易把这套规范理解成“必须全收下”，而不是“可渐进启用”。

**建议**：

1. `init` 改成按配置装配文件集，而不是全量复制后裁剪。
2. 默认只安装基础入口 + 当前语言规范 + 当前项目类型对应验收文档。
3. 其余治理文档通过 `add skill` 或 `add skill-pack` 按需补装。

### 问题 3：`skills.lock.json` 还不是同步与诊断的真实配置源

**优先级**：高

**现象**：

- `skills.lock.json` 已记录 `includeClaude`、`includeDocsGovernance`、`projectType`、`codeIndexGranularity` 等关键信息。
- 但 `sync` 主要还是按模板全量遍历文件，再结合少量排除规则决定是否同步。
- `doctor` 对必需文件的判断也没有完整依赖这些配置位。

**影响**：

- 项目初始化后的真实选择，后续不一定能被 `sync` 和 `doctor` 正确保留。
- 例如关闭了某些能力，后续同步仍可能把模板文件重新补回来。
- 长期看，配置状态、文件状态、CLI 诊断状态会逐渐分叉。

**建议**：

1. 把 `skills.lock.json` 明确提升为“项目已启用规则的权威来源”。
2. `sync` 基于 lock 计算托管文件集合，而不是单纯遍历模板目录。
3. `doctor` 的必需文件、可选文件、告警项都应根据 lock 动态调整。

### 问题 4：代码索引规则设计偏重，但 CLI 没有提供足够自动化支撑

**优先级**：中

**现象**：

- `init` 会询问代码索引粒度，并写入 `skills.lock.json`。
- 模板要求在启用后维护 `developers/CODE-INDEX.md`。
- 但 CLI 目前不会生成索引文档，只会在 `doctor` 中给出缺失告警。

**影响**：

- 代码索引成为一个高维护成本、低自动化支撑的规则点。
- 在真实项目里，这类文档最容易快速过期，最终失去可信度。

**建议**：

1. 最低限度提供 `doctor --fix` 或新命令来生成 `CODE-INDEX.md` 骨架。
2. 如果短期内不做自动化，建议把代码索引能力明确降级为高级可选能力，不作为常规初始化项。
3. 将“是否启用代码索引”从默认问答中后移，减少首次接入心智负担。

### 问题 5：`init` 的自动化与安全性体验还不够稳

**优先级**：高

**现象**：

- `options.js` 支持 `--target`，但 `runInit` 实际并未真正按该值作为初始化目标。
- `--yes` 模式直接使用当前工作目录。
- 初始化复制阶段默认 `force: true`，已有文件会被直接覆盖。

**影响**：

- 批量接入、CI、脚本化接入都不够顺手。
- 对已有项目执行初始化时，默认覆盖行为存在实际风险。

**建议**：

1. `init` 必须统一尊重 `--target`。
2. 默认改为“发现已存在文件时提示并中止”或进入预览模式。
3. 将“强制覆盖”单独设计为显式开关，而不是当前默认行为。

### 问题 6：模板命令口径不完全统一，影响第一次上手

**优先级**：中

**现象**：

- 模板内多份技能文档使用 `npx @skybluefeet/skills-kit ...`。
- CLI 帮助、部分诊断提示则使用 `project-rules-kit ...`。

**影响**：

- 新用户第一次接入时，需要先理解“包名”和“bin 名”之间的关系。
- 文档示例看起来像两套命令体系，增加理解成本。

**建议**：

1. 统一一套主推荐写法。
2. 另一种写法作为补充说明保留在接入文档或 FAQ，而不是分散在多个技能文档里。

---

## 5. 优先级改进顺序

### 第一优先级

1. 让 `includeDocsGovernance` 对应的模式真正自洽。
2. 让 `skills.lock.json` 成为 `sync` / `doctor` 的真实配置源。
3. 修正 `init` 的 `--target` 与默认覆盖行为。

### 第二优先级

1. 将 `init` 从“全量复制后裁剪”改为“按配置装配模板”。
2. 收缩默认安装的技能集合，减少首次接入体积。

### 第三优先级

1. 决定代码索引能力是继续工程化还是降级为高级可选能力。
2. 统一模板中的命令示例口径。

---

## 6. 建议的改造方向

如果后续进入实施阶段，建议按以下顺序落地：

1. 先改 `init`、`sync`、`doctor` 对 `skills.lock.json` 的读取逻辑，建立统一配置真源。
2. 再把模板入口拆成可配置模式，解决轻量模式与完整模式互相打架的问题。
3. 最后处理代码索引和命令口径这类增强项，降低首次接入和长期维护成本。

---

## 7. 最终结论

从实际使用角度看，这套模板规范已经具备良好的结构基础，但当前 CLI 还没有把“不同治理强度、不同安装模式、不同配置状态”的差异完整落地。真正需要优先修正的不是增加更多规则，而是让现有规则在初始化、同步、诊断三个关键流程里保持一致。

只要先解决“模式自洽”和“lock 真源”这两个问题，这套模板的实际可用性会明显提升。
